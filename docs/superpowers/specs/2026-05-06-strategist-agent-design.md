# Agent 1 — Creative Strategist Memory Agent

**Date:** 2026-05-06
**Status:** Design approved, ready for implementation plan
**Scope:** Agent 1 only (Strategist memory layer). Agent 2 (Producer) is a separate future spec.

---

## 1. Purpose

A worker that reads every judged ad task in the system, synthesizes patterns across winners and losers per product, and maintains a living "strategist memory" that future agents (Agent 2) and humans read. It is the long-term creative-knowledge moat for the team.

The strategist memory captures what a senior creative strategist *and* an editor would need to make new winning creative: angles, personas, formats, hooks, lingo, visual direction, production specifics, and the loser patterns to avoid.

## 2. Goals

- Synthesize qualitative patterns from existing ClickUp + Matrix data without requiring new manual labeling
- One memory artifact per product (no cross-product blending)
- Live, browsable in the app via a new Strategist tab
- Auto-refresh once a day when the dashboard is open, plus a manual refresh button
- Rebuilds incrementally so daily runs are cheap on rate limits
- Produces both a structured JSON (for Agent 2 programmatic consumption) and a narrative markdown (for humans)
- Snapshots to repo so we get git-diff history of how the strategy evolved
- Designed to absorb Meta/Google performance data later without rewriting

## 3. Non-Goals (Out of Scope)

- **Agent 2 (Producer)** — the image-ad generation pipeline that consumes strategist memory. Separate spec.
- **Performance data ingestion** — Meta Ads / Google Ads spend, ROAS, CTR, hook rate. Schema reserves fields, ingestion comes later.
- **Cross-product synthesis** — "what wins across all products" view. Each product is its own memory.
- **New ClickUp custom fields or workflow changes** — the strategist consumes existing data only.
- **Real-time on-event re-synthesis** — daily + manual is sufficient for v1.

## 4. Existing Infrastructure Leveraged

The strategist slots into existing systems with no new daemons or product management:

| System | How it's used |
|---|---|
| `products` table in Supabase | Source of products + their `clickup_list_id` |
| `activeProductId` global / `localStorage.active_product` | Drives which product's memory the UI shows |
| `tools/classify_worker.py` worker fleet | Extended to handle a new `strategist` task type — no new daemon |
| `claude -p` CLI in headless mode | Same invocation pattern as existing classify worker. Uses Max subscription, $0 marginal cost |
| `auto_pause_when_claude_idle` config | Inherited — strategist worker pauses when user is actively coding |
| Supabase Realtime channels (`cc-${productId}`) | Strategist tab subscribes to memory-row updates per product |
| Existing tab + ap-table + pulse strip CSS | Strategist tab mirrors Action Plan polish |

## 5. Data Sources (read-only)

For every judged ad task, the strategist reads:

- **ClickUp task fields**: title, description, status, custom fields (see §5.2 below for the canonical schema)
- **ClickUp task comments**: full conversation history; often contains the real reasoning ("we changed talent because…")
- **ClickUp linked doc page**: the 7-section creative brief generated by the classify-inspiration skill (when present)
- **Supabase Matrix data**: cell coordinates, parent inspiration ID (if any), parent ad ID (if it's a variation)

Tasks scoped per product via `products.config->>'clickup_list_id'` (JSONB field on `products` row).

### 5.1 Status filter — confirmed by reading [immuvi-command-center.html:36945-36947](immuvi-command-center.html:36945)

Lists use lowercase ClickUp status names. Worker matches case-insensitively.

| Group | Status values | Treatment |
|---|---|---|
| **Winners** | `winner`, `mild winner`, `scale`, `complete` | Synthesize "why it won" briefs, contribute to `winning_patterns` counts |
| **Losers** | `loser`, `killed` | Synthesize "why it died" briefs, contribute to `losing_patterns` counts |
| **Excluded** | `untested`, `approved`, `in production`, `ready to launch`, `testing` | No verdict yet — skip |

### 5.2 Canonical custom field schema — confirmed via ClickUp API on list 901613534733

These field names + dropdown options are standardized across product lists. The strategist must use these exact taxonomies (not invent its own):

| Field | Type | Canonical values |

| **Production Style** | drop_down (10) | Organic / Raw UGC · Polished UGC · Professional Studio · AI Generated · Screen Record · Animation / Motion · Static Graphic · Slideshow · Repurposed Organic · Competitor Inspired |
| **Funnel Type** | drop_down (3) | TOF · MOF · BOF |
| **Photo/Video** | drop_down (3) | Photo · App Dev · Video |

| **Spend** | number | $ spent on the ad |

Worker fetches custom field definitions per list at start of each run (one API call) and maps task field values to these canonical names. Lists where a field is renamed or absent fall back to inferring from description + comments.

### 6.2 Markdown narrative (`strategist_memory.md` / `markdown` Supabase column)

A senior strategist's memo, generated from the JSON. Sections:

1. **Header** — product name, last updated, run stats
2. **What's winning right now** — top 5 angles, personas, formats, hooks
3. **Combinations to bet on** — proven angle×persona×format×hook combos
4. **Vocabulary that wins** — verbatim hooks, lingo, CTAs
5. **Production playbook** — talent, setting, props, lighting, composition recipes (the editor-facing section)
6. **What's dying / don't bother** — loser patterns and why
7. **Trends** — rising / fading in last 14 days
8. **Full winners log** — every winner brief in narrative form
9. **Full losers log** — every loser brief in narrative form

## 7. Storage

### 7.0 Existing schema reminder (read-only for the strategist)

The `products` table already exists with shape:
```
products(id text PK, name text, config jsonb, created_at, updated_at)
```
The ClickUp list ID lives at `products.config->>'clickup_list_id'`. Worker queries:
```sql
select id, name, config->>'clickup_list_id' as clickup_list_id
from products
where config->>'clickup_list_id' is not null;
```

### 7.1 New Supabase tables

```sql
-- One row per product. Source of truth for memory.
create table strategist_memory (
  product_id   text primary key references products(id) on delete cascade,
  json         jsonb       not null,
  markdown     text        not null,
  updated_at   timestamptz not null default now()
);

-- Run queue + log. Realtime-subscribed by Strategist tab for live progress.
create table strategist_runs (
  id              bigserial primary key,
  product_id      text not null references products(id) on delete cascade,
  status          text not null check (status in ('pending','running','done','failed')),
  trigger         text not null check (trigger in ('daily','manual','backfill')),
  run_date        date not null,                -- for daily-dedup unique constraint
  started_at      timestamptz,
  finished_at     timestamptz,
  tasks_processed int  default 0,
  tasks_skipped   int  default 0,
  error           text,
  created_at      timestamptz not null default now()
);

-- Daily-dedup: one daily row per product per day
create unique index strategist_runs_daily_dedup
  on strategist_runs(product_id, run_date)
  where trigger = 'daily';

-- Per-task synthesis cache, so daily runs only re-process changed tasks.
create table strategist_processed (
  product_id        text not null references products(id) on delete cascade,
  clickup_task_id   text not null,
  content_hash      text not null,              -- hash of (status + description + comments + custom fields)
  brief_json        jsonb not null,             -- the per-task winner_brief / loser_brief entry
  is_winner         boolean not null,           -- true for Winner/Mild Winner/Scaled, false for Loser/Killed
  last_synthesized  timestamptz not null default now(),
  primary key (product_id, clickup_task_id)
);
```

RLS: open `anon` access on read for `strategist_memory` and `strategist_runs` (matches existing app pattern). Write access on insert to `strategist_runs` from anon (for queueing). Worker uses service role.

### 7.2 Repo snapshot

After each successful run, the worker writes:
- `tools/strategist_memory/<product_slug>.md`
- `tools/strategist_memory/<product_slug>.json`

`product_slug` = slugify(`products.name`) (lowercase, hyphens, ASCII).

User commits these manually when they want a checkpoint. No auto-commit (worker has no git auth).

## 8. Pipeline (one worker run)

For a `strategist` task in the worker queue:

1. **Resolve product** — load `products` row by `product_id`, fail if no `clickup_list_id`
2. **Fetch judged tasks** — ClickUp API: list all tasks in `clickup_list_id` filtered to status ∈ {Winner, Mild Winner, Scaled, Loser, Killed}
3. **Diff against cache** — for each task, compute `content_hash` of (status + description + comments + custom fields). Compare to `strategist_processed.content_hash`. Only process where hash differs or row absent.
4. **Per-task synthesis** — for each new/changed task:
   - Fetch full task detail (description, comments, custom fields, linked doc page if any)
   - Bundle into a synthesis prompt
   - `claude -p` headless → returns the per-task `winner_brief` / `loser_brief` JSON entry
   - Upsert `strategist_processed`
5. **Aggregate** — load *all* `strategist_processed` rows for this product. Roll up dimension counts (angles, personas, formats, hooks, lingo, etc.). Compute `combinations_that_win`, `combinations_that_die`, `trends.rising`, `trends.fading`.
6. **Render markdown** — single Claude call: pass the aggregate JSON, ask for the strategist memo in the structure from §6.2
7. **Upsert `strategist_memory`** — write JSON + markdown + `updated_at`
8. **Write repo snapshot** — `tools/strategist_memory/<product_slug>.{md,json}`
9. **Update `strategist_runs`** — `status='done'`, `finished_at`, `tasks_processed`

## 9. Trigger Model

### 9.1 Daily auto-run (dashboard-driven, runs only when app is open)

The daily check runs **only for the active product** — products no one is looking at don't burn rate-limit quota.

Trigger points:
- On dashboard load (after `activeProductId` resolves)
- On product switch (every time `activeProductId` changes)

Logic:
```
on activeProductId set/change:
  if no strategist_runs row where product_id = activeProductId
                            and run_date = today
                            and trigger = 'daily':
    INSERT (product_id=activeProductId, status='pending', trigger='daily', run_date=today)
```

Unique partial index on `(product_id, run_date) where trigger='daily'` prevents multi-tab double-queue.

Net effect: every product you actually use gets refreshed at most once per day, the first time you look at it that day.

### 9.2 Manual refresh

"Refresh Strategist Memory" button in Strategist tab:

```
INSERT (product_id=activeProductId, status='pending', trigger='manual', run_date=today)
```

Manual rows skip the daily-dedup unique constraint (different `trigger` value).

### 9.3 Initial backfill

First run for a never-processed product is identical to a normal run — the diff against cache is empty, so all judged tasks get synthesized. This happens automatically the first time the dashboard loads with a product selected. No special button.

Estimated wall-clock: ~10–20 min per product on first run, sequential `claude -p` calls. $0 marginal cost (Max subscription).

## 10. Worker Integration

### 10.1 Existing worker (`tools/classify_worker.py`)

Currently polls one task type. We add a second task type: `strategist`.

Approach: factor out the task-dispatch into a registry. The poll loop reads the queue (now: a unified queue or two parallel polls), dispatches by `task_type`. Existing `classify` task type unchanged.

### 10.2 Polling

The worker polls **two** queues each cycle:
1. Existing inspiration classify queue
2. New `strategist_runs` table where `status='pending'` ordered by `created_at`

Both run sequentially — never two `claude -p` calls in flight at once. Honor `auto_pause_when_claude_idle`.

### 10.3 Worker file layout

- `tools/classify_worker.py` — main loop, task dispatch, claims
- `tools/strategist_pipeline.py` — new module with the strategist pipeline (steps 1–9 above)
- `tools/clickup_client.py` — shared ClickUp API helper if not already extracted

## 12. Failure Handling

- **Per-task synthesis fails**: log task ID + error to `strategist_runs.error`, increment `tasks_skipped`, continue with the rest. Bad tasks don't block the run.
- **Aggregate / markdown render fails**: mark run `failed`, leave previous `strategist_memory` row untouched. UI shows red pill: "Last run failed — retry".
- **ClickUp rate limit**: exponential backoff (1s → 2s → 4s → max 60s). Same pattern as existing classify worker.
- **Worker crash mid-run**: claimed `running` rows are reset to `pending` on next poll if `started_at > 30 min ago` (stale-claim recovery).
- **Repo snapshot fails**: log warning, do not fail the run. Supabase row is the source of truth.

## 13. Rate-Limit Guardrails

- All `claude -p` calls sequential, never parallel
- Worker honors `auto_pause_when_claude_idle` (pauses when user is actively coding)
- Daily incremental delta is small (only changed tasks); first-run backfill is a one-time hit
- Spread one product's run across ~10–20 min on first run; subsequent runs are sub-minute

## 14. Future Hooks

These are explicitly designed-for-later, not implemented now:

- **Performance data layer** — Spend + Revenue + ROI are already computed in v1 from existing custom fields. Reserved fields for v2 (when Meta/Google API ingestion lands): `cpa`, `roas`, `ctr`, `hook_rate`, `hold_rate`, `cpm`, `frequency`. No schema rewrite needed.
- **Agent 2 read interface** — Agent 2 worker will `SELECT json, markdown FROM strategist_memory WHERE product_id = X` and stuff both into its prompt. No API layer needed.
- **Cross-product synthesis** — could be added as a separate "global strategist" memory aggregating across products. Not for v1.
- **Trend diff view in UI** — show "what changed since yesterday" using git history of repo snapshots. Not for v1.

## 15. Open Questions / Risks

- **Comment volume**: tasks with 50+ comments could blow context windows. Mitigation: cap at last 30 comments; spec is fine for v1.
- **Linked doc pages discovery**: ClickUp tasks don't always link the brief doc explicitly. Worker checks task description for ClickUp doc URLs and fetches if found. Best-effort.
- **Concurrent manual + daily**: two pending rows for same product. Worker processes in `created_at` order, both succeed but second has no work after the first (everything's in cache). Acceptable.
- **Spend/Revenue data quality**: not every task has Spend + Revenue filled in. Strategist computes ROI only when both > 0; otherwise flags `spend_data_complete: false` in the brief and falls back to qualitative-only patterns for that task.
- **Persona/Angle dropdowns currently empty on inspected list** — primary signal is the free-text `Persona Tag` / `Angle Tag` short_text fields. Worker reads the Tag fields first, falls back to dropdown values when populated. Future cleanup: backfill dropdowns from Tag fields (out of scope).
